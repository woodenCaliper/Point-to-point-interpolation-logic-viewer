# AI Implementation Instructions

## 現在の運用モード
- 現在は**実装フェーズ**。
- `human requirement.md` の要件に従って `index.html` を更新し、差分を検証して進める。

## 要件サマリ（`human requirement.md` 準拠）
- 単一HTMLで 2D 経路比較ビューアを実装する。
- 必須アルゴリズムは Linear / Dubins / CC-Dubins / Single Clothoid G1。
- 単位は位置系=mm、角度系=°（degree）。
- 数値入力は整数のみとする（小数入力なし）。
- 最短判定は距離のみで行う。
- NG値は入力拒否（バリデーションエラー時は生成計算に反映しない）とする。
- NG値入力時は当該フィールドを赤枠で可視化する。
- NG値は入力欄に残したまま保持し、その間は経路再計算を停止する。
- 左ペインは入力専用、右ペインは表示設定・結果表示専用。
- 候補パスの標準並び順はアルゴリズム固有順とする。
- パス生成は入力変更の即時反映とし、生成ボタンは設置しない。
- サンプリング間隔は 1mm 固定とし、ユーザー入力で変更させない。
- CC-Dubins は厳密方式で実装する（近似実装は不可）。
- 単一クロソイドG1補間は、始点・終点の位置と方位（G1）を満たす単一セグメントのクロソイドとして実装する。
- CC-Dubins の参照ソースは Holger Banzhaf らの `steering_functions` とする。
- GitHub Pages で公開し、ブラウザのみで利用可能にする。

## UI実装方針（実装再開時に適用）
1. **左ペイン構造**
   - アルゴリズム選択UIを最上部に配置。
   - 入力フォームを Start point / End point / Path の3セクションに固定。
   - 選択アルゴリズムに応じて各セクション内の入力を表示切替する。

2. **単位ポリシー**
   - `x, y, R` など長さ関連は mm で表示。
   - `θ` は degree で入力し、内部計算時に rad に変換。
   - 曲率 `k` は `1/mm` として表示。

3. **中央ペイン始点アイコン表現**
   - `xy`: 点のみ描画。
   - `xyθ`: 点 + 向き矢印。
   - `xyθk`: 点 + 向き矢印 + 曲率半径を示す円弧ガイドを描画。
   - 座標系はワールド基準で「右 +X / 上 +Y」に統一する。

4. **右ペイン責務**
   - グリッドは常時表示とし、サンプル点・曲率ガイド等の表示トグルを提供（向き矢印は常時表示）。
   - 経路長、セグメント列、サンプル点数、選択経路種別を表示。
   - 複数候補パスがある場合に備え、候補ごとの表示ON/OFFトグルを提供。
   - 候補リストの表示順はアルゴリズム固有順を維持する。
   - 「最短距離のみ表示」ボタンを提供し、最短候補のみ可視化状態へ一括切替。
   - 同率最短時は先頭候補1件のみを最短候補として扱う。
   - ON中は個別トグルをdisabled化し、視覚的に灰色表示。
   - ON前にoffだった項目は `☐` のまま表示。
   - ON前にonだった項目は `-`（indeterminate）表示。
   - OFF時はON前スナップショットの可視状態へ復元。

5. **入力イベント方針**
   - 各入力変更イベント（`input` / `change`）で即時に再計算・再描画する。
   - 明示的な「Generate」ボタンは置かない。
   - reject型バリデーション時は赤枠クラスを付与し、正常化時に解除する。
   - バリデーションNG時は入力値を保持しつつ、再計算・再描画をスキップする。

6. **視点操作 / 補助表示**
   - ホイールでカメラスケールを変更（カーソル中心ズーム）。
   - 左ドラッグでカメラ中心を移動（パン）。
   - キャンバス左下に固定座標軸ギズモ（+X / +Y）を描画。

7. **グリッド分解能ルール**
   - スケールに応じてグリッド間隔を動的変更。
   - 間隔は 1 または 5 の有効数字系列（`1, 5, 10, 50, ...`）から選ぶ。
   - 現在のグリッド間隔（mm）を中央ペインにオーバーレイ表示する。
   - 表示位置は座標軸ギズモ（+X / +Y）付近に配置する。

8. **CC-Dubins 厳密実装方針**
   - CC-Dubins は採用文献・定義式・拘束条件を先に固定し、その仕様に一致する実装のみ許容する。
   - 参照実装/定義は Holger Banzhaf らの `steering_functions` を正とする。
   - 近似的な補間や暫定実装は導入しない。

9. **公開方針（GitHub Pages）**
   - 配布形態は GitHub Pages を前提とする。
   - 公開手順は GitHub 公式ドキュメントで一般的な方法（Pages設定または公式Actions）を優先する。
   - 単一HTMLで完結し、追加ビルドなしで公開可能な構成を維持する。
   - 実装再開後は Pages 上で動作確認し、URLをREADMEに明記する。

10. **性能・精度方針**
   - 性能目標は厳格にしすぎず、1回のパス生成を 0.1 秒以内（目安）とする。
   - 数値誤差処理は過度に厳密化せず、実装容易性を優先したシンプルな方式を採用する。
   - 最短判定は参考値レベルのため、保守しやすい簡易判定ロジックを優先する。

11. **候補リスト命名方針**
   - 命名可能な候補（例: `RSR`, `LSL` など）はアルゴリズム由来の名称を表示する。
   - 命名が難しい候補は `1, 2, 3 ...` の連番ラベルを付与する。

## 状態管理方針（実装再開時）
- `paths[]`: 生成候補パス配列（id, label, length, visible, points）。
- `shortestPathId`: 距離最小の候補IDを保持。
- 同率の場合はアルゴリズム固有順で先頭の候補IDを `shortestPathId` に採用。
- 個別トグル操作は `paths[i].visible` を更新。
- `beforeShortestOnlyVisible[]`: 最短のみ表示ON前の可視状態スナップショット。
- 「最短距離のみ表示」ON時は `visible = (id === shortestPathId)` を一括適用し、個別トグルをdisabled化。
- OFF時は `beforeShortestOnlyVisible[]` から可視状態を復元。
